What a Test Plan Can't Tell You

by Jeff Thoensen

A written test plan tells you a feature works the way it was specified. It doesn't tell you what happens when someone uses it the way people actually do, which is rarely the way it was specified.

I keep a bulk-edit feature as the example I come back to. The test plan covers selecting a few rows, editing a field, saving, and confirming the change. Every case passes. During an exploratory pass afterward, I select a few hundred rows instead of a few, because that's closer to how someone would actually use a bulk-edit feature once they trust it. The save button spins and never resolves. Nothing in the UI indicates anything went wrong. The backend had a request size limit that silently dropped the request past a few dozen rows.

None of that shows up in a plan written before anyone's touched the feature, because the plan describes intended behavior, and this bug only exists at the edges of what the feature was actually built to handle. Exploratory testing works by chasing those edges instead of confirming the middle.

I still start most sessions with a written plan, since it catches the obvious regressions fast and gives me a baseline to compare against. What the plan misses is the stuff a real person would stumble into by accident: pasting in a value with a trailing space, refreshing mid-submit, opening the same record in two tabs, resizing a window while a modal is open. None of that is exotic. It's just outside what anyone thought to write down.

The way I run a session is time-boxed, usually thirty minutes, with a rough charter instead of a script. Something like "try to break the bulk-edit save path" is enough of a starting point. I take notes as I go, not after, since the interesting moments are usually small and easy to forget by the time the session ends: a spinner that took a beat too long, a field that reset itself after a validation error, a count that was off by one after a delete.

The bulk-edit bug was small in isolation, but it's the kind of bug that erodes trust in a product fast: the person hitting it has no idea anything failed, and just finds out later that their data isn't there.

I've had engineers ask why this needs a person at all, since a large enough automated suite should eventually hit the same edge cases through sheer volume. Sometimes it does, though an automated suite only tests what someone already thought to write an assertion for. A person poking around with intent finds the untested edges faster, and can follow a hunch the moment something looks slightly off.
